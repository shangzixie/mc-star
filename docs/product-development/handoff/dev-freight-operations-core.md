# Development Handoff Spec - Freight Operations Core

## Goal

把发货管理主链路落成一个内部运营后台，先保证数据正确和操作清晰，再逐步补强细节能力。

## Scope

- 适用角色：业务、客服、仓库、管理
- 所属阶段：MVP 核心链路
- 相关能力：入库、库存、发货单、柜、出货、附件

## Delivery Surface

- 页面：内部管理后台 freight 相关页面
- API：`/api/freight/**`
- 数据：`src/db/schema.ts` 对应 freight 相关表

## Preconditions / State Requirements

- 主数据先存在：合作方、运输节点、仓库
- 入库后才能形成库存批次
- 出货必须基于库存批次
- 最终扣库存只能发生在明确出库动作

## Input

必填：
- 业务单号 / 入库单号等业务标识
- 对应主数据主键
- 数量字段

可选：
- 联系资料
- MBL 信息
- 员工分配
- 附件

## Validation Rules

1. 数量相关校验必须放在后端
2. 出货状态必须单向推进
3. 任何库存扣减都必须可回溯到流水或分配记录
4. 前端不得绕过后端约束自行认定库存可用量

## UX Notes

1. 非技术人员优先，避免复杂多层弹窗和隐藏入口
2. 创建流程优先最小必填，再在详情页补录
3. 列表页应优先提供搜索、状态筛选、查看详情
4. 详情页应按业务区块拆分，而不是一次展示全部字段

## Acceptance

1. 用户可以完成入库、查看库存、推进出货
2. 库存数量变化可追踪
3. 页面交互不要求技术背景即可理解
4. 新功能进入实现前，能在 `coding/` 找到对应技术事实
